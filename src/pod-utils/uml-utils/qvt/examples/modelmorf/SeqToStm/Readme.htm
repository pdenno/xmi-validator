Mapping between sequence diagrams and state machines

------------------------------------------------------------------------

 

Folder ‘spec’ contains:

SeqMM.xml : Sequence Diagram Meta Model

StmcMM.xml : State Machine Meta Model

SeqUM.xml : A sample sequence diagram model

StmcUM.xml : A sample state machine Model

SeqToStm.qvt : QVT specification file for sequence diagram to state machine mapping.

 

 

Sequence Diagram Meta Model :

A sequence diagram specifies an interaction scenario. An interaction consists of a set of lifelines with message flows between them. A message flow begins with a send message event on a lifeline and ends with a receive message event on another (or same) lifeline. Message flow sequencing is specified by means of previous-next associations between message events. A lifeline may annotate a message event with the state the system

is expected to be in when the event occurs.

 

See figure 'Simple Meta Model for Sequence Diagram for example SeqToStm' in file SeqToStm.pdf

 

 

State Machine Meta Model :

The state machine model is straight-forward : there are states and transitions. A transition may have a guard, an event trigger and an action.

Message sending is one such action.

 

See figure 'Simple Meta Model for State Machine for example SeqToStm' in file SeqToStm.pdf

 

 

The transformation maps :

  • Each state of the interaction to a state of the state machine
  • Each message to a transition between states corresponding to the states that annotate the send-receive events
  • Each adjacent message-flow pair to a pair of adjacent transitions that map to the corresponding messages

 

 

The sample sequence diagram model given in file SeqUM.xml contains:

  • Interaction :
    • Door
  • LifeLine :
    • Opened
    • Closed
    • Locked
  • State :
    • Opened : lifeline=Opened
    • Closed : lifeline=Closed
    • Locked : lifeline=Locked
  • Message :
    • CloseDoor
    • LockDoor
  • MessageEvent :
    • Send : message=CloseDoor; state=Opened; type='Send'
    • Receive : message=CloseDoor; state=Closed; type='Receive'
    • Send : message=LockDoor; state=Closed; type='Send'
    • Receive : message=LockDoor; state=Locked; type='Receive'

 

See figure 'Sequence Diagram model for example SeqToStm' in file SeqToStm.pdf

 

 

The sample state machine model given in StmcUM.xml contains:

  • StateMachine :
    • Door
  • State :
    • Opened
    • Closed
    • Init
  • Transition :
    • OpenDoor : fromState=Closed; toState=Opened; trigger=Open; action=OpenDoor
    • CloseDoor : fromState=Opened; toState=Closed; trigger=Close; action=CloseDoor; guard=doorWay->isEmpty
    • Create : fromState=Init; toState=Opened
  • Message :
    • OpenDoor
    • CloseDoor

 

 

The following inconsistencies exist between the two models:

  • State ‘Locked’ does not exist in StateMachine ‘Door’.

 

 

If the example is run in checkonly mode following inconsistency will be reported:

State ‘Locked’ does not exist in StateMachine ‘Door’ corresponding to State ‘Locked’ on LifeLine ‘Locked’ of Interaction ‘Door’.

 

To run this example in checkonly mode execute the batch file ‘run1.bat’ for windows or  the shell script ‘run1.sh’ for unix.

 

 

If the example is run in enforce mode, enforcement removes the inconsistency by creating a new State 'Locked', a new Transition 'LockDoor' and a new ‘SendMsg’ action to send the Message 'LockDoor'.

 

See figure 'State Machine model for example SeqToStm' in file SeqToStm.pdf

 

To run this example in enforce mode execute the batch file ‘run2.bat’ for windows or  the shell script ‘run2.sh’ for unix.
